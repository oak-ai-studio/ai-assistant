## 模块一：用户与助理配置

### 1.1 Onboarding 流程

Onboarding 的核心原则：**最低摩擦进入，助理通过对话自然收集信息，而非让用户填表。**

#### 1.1.1 流程步骤

1. **欢迎页**：展示产品价值主张，一句话说清楚「这是一个越用越懂你的AI助理」，提供「开始使用」按钮
2. **给助理起个名字**（可跳过，默认名「小助」）
3. **选择技能**（必选一个，MVP 提供：背单词 / 做饭助理 / 随便聊聊）
4. **技能初始配置**（步骤3.5，见下方详述）
5. **直接进入首页**，助理发起第一条消息主动了解用户

#### 1.1.2 步骤3.5：技能初始配置

> **决策来源**：Boss 确认采用「方向B」，在 Onboarding 中加入技能初始配置屏，而非完全依赖进 App 后的对话收集。

**规则：**

- 用户选完技能后，对每个已选技能展示 **1 个最关键的偏好问题**（每个技能各 1 屏）
- 回答直接写入记忆系统，作为该技能的初始偏好
- 每个技能只问 1 个问题，保持轻量
- 可跳过——跳过后由助理在后续对话中自然收集

**各技能示例问题：**

| 技能 | 配置问题 | 选项示例 |
|------|---------|---------|
| 背单词 | 你希望以什么方式记忆单词？ | 通过场景例句 / 填空练习 / 词语翻译 |
| 做饭助理 | 你平时的烹饪水平是？ | 新手入门 / 会做基础家常菜 / 有一定经验 |
| 随便聊聊 | 你希望助理聊天时的风格是？ | 轻松有趣 / 温暖治愈 / 直接理性 |

**验收标准：**

- Given 用户在步骤3选择了一个或多个技能
- When 点击「下一步」进入步骤3.5
- Then 每个已选技能依次展示 1 个配置屏，用户可回答或跳过
- And 回答内容以 `preference` 类型写入记忆系统，`skill_source` 标记为对应技能
- And 跳过后该技能不写入初始偏好，进度正常推进

---

#### 1.1.3 冷启动对话设计

用户跳过配置后，助理首条消息需承担信息收集职责，示例：

> 「你好！我是你的私人助理，我会在和你聊天的过程中慢慢了解你。」
>
> 「先简单问你一下——你学英语主要是为了工作、考试，还是日常兴趣？」
>
> （根据回答继续追问，自然收集偏好，写入记忆系统）

---

### 1.2 助理配置

用户可通过首页右上角菜单进入「配置助理」页面：

| 配置项   | 说明                             |
| ----- | ------------------------------ |
| 助理名字  | 用户自定义，支持随时修改                   |
| 助理头像  | MVP 阶段提供 6 个预设头像选择             |
| 背景提示词 | 用户用自然语言描述助理风格，例如「严格一点，督促我完成任务」 |
| 已安装技能 | 显示当前已激活的技能列表，可调整顺序、停用技能        |
| 推送时间  | 设置助理每日主动触达的时间段                 |

---

### 1.3 技能管理

技能以插件形式存在，每个技能包含以下属性：

| 属性 | 说明 |
|------|------|
| 技能名称 | 显示在首页看板的名称，用户可自定义 |
| 技能图标 | 首页展示用 |
| 专属 System Prompt | 进入该技能对话时注入，覆盖全局 System Prompt 的技能部分 |
| 看板页面 | 该技能对应的独立功能页面（如单词列表、食谱列表） |
| 今日待办权限 | 是否允许该技能向首页待办卡片写入条目 |

**MVP 内置技能清单：**

- 背单词：词汇管理、每日学习、测验
- 做饭助理：食谱管理、食材追踪、做饭引导
- 随便聊聊：无特定目标的日常对话，主要用于记忆积累

## 模块二：技能插件与首页看板

### 2.1 首页结构

> **导航方案更新（v1.1）**：Boss 确认采用方案B，去掉底部导航，改用 FAB（悬浮按钮）作为全局对话入口。

首页是所有技能的统一入口，由以下区域组成：

#### 2.1.1 全局对话入口：FAB（悬浮按钮）

| 属性 | 说明 |
|------|------|
| 位置 | 首页右下角固定悬浮 |
| 样式 | 圆形 52px，橙色实心，白色对话图标 |
| 功能 | 点击唤起全局对话（不限定技能的对话入口） |
| 交互 | 点击后进入对话页面，顶部可切换技能上下文 |
| 可见性 | 任何页面均可见（全局悬浮） |

**FAB 上下文感知机制**

> **核心设计（Boss 确认）**：FAB 不仅是对话入口，更是全局上下文感知的智能助理唤起点。

- FAB 在所有页面底部固定悬浮（首页/记忆列表/技能页面/设置页面）
- 点击 FAB 唤起对话时，AI 能感知用户当前所在页面和页面内容
- 根据上下文提供相关对话开场白：
  - **在记忆列表页** → "要不要我帮你整理一下记忆？"
  - **在背单词页面** → "今天学了几个词？我来考考你"
  - **在做饭页面** → "这道菜需要什么食材？"
  - **在首页** → 通用问候

**技术实现要点：**
- 页面上下文通过标准化 JSON 注入（见 `3.3 页面上下文注入`）
- 对话开场白由 LLM 基于当前页面动态生成，不使用固定模板
- 用户可忽略开场白，直接输入自己的问题

**导航结构：**
- **首页**：技能看板（主页面）
- **对话**：通过 FAB 唤起
- **记忆**：右上角菜单
- **设置**：右上角菜单（配置助理）

**设计理由**：简化界面，减少视觉噪音，对话通过 FAB 快速唤起更符合"助理随时待命"的产品调性。

---

#### 2.1.2 顶部问候区

| 元素 | 说明 |
|------|------|
| 问候语 | 根据当前时段动态生成：上午好 / 下午好 / 晚上好，加上助理名字 |
| 今日待办卡片 | 汇总各技能写入的待办条目，最多显示 3 条，点击跳转对应技能页 |
| 右上角菜单 | 下拉展示「配置助理」和「记忆」两个入口 |

#### 2.1.3 技能看板区

以 2 列网格展示已安装技能，每个技能卡片包含：

- 技能图标 + 名称
- 一行状态摘要（如「今日已学 15 个单词」「有 3 个食谱待尝试」）
- 右下角红点（有未处理待办时显示）

用户可长按技能卡片拖拽排序，排序结果持久化存储。

#### 2.1.4 今日待办写入规则

| 规则 | 说明 |
|------|------|
| 写入时机 | 每个技能的后台任务每日定时执行，检测是否需要生成待办 |
| 写入内容 | 自然语言描述，例如「今天还有 20 个单词没有复习」 |
| 优先级 | 技能可标记待办优先级（高/中），首页按优先级排序显示 |
| 最大数量 | 每个技能每日最多写入 2 条待办，避免首页过于拥挤 |

## 模块三：对话系统

### 3.1 对话入口

对话系统有两个入口：

| 入口 | 说明 |
|------|------|
| 全局 FAB | 见 `2.1.1 全局对话入口：FAB`，任何页面可见，点击唤起全局对话 |
| 技能内对话按钮 | 各技能看板页内的对话入口，点击唤起时自动带入该技能上下文 |

---

### 3.2 半屏对话框设计

对话框以半屏抽屉形式出现，叠加在当前页面之上，与底层页面解耦：

- 支持上滑展开全屏，下滑收起
- 收起后不中断对话，重新打开恢复当前会话
- 键盘弹出时对话框自动上移，不遮挡输入框
- 底层页面内容轻度模糊处理，聚焦对话

---

### 3.3 页面上下文注入

每个页面模块定义标准化的上下文 JSON，在对话请求时附加到 System Prompt：

```json
{
  "current_page": "vocabulary_list",
  "skill": "english_learning",
  "data": {
    "today_learned": 15,
    "today_target": 20,
    "current_word": { "word": "albeit", "meaning": "尽管" }
  },
  "user_action": "viewing_word_detail"
}
```

LLM 收到上下文后可自然结合页面信息回应，例如用户问「这个词怎么用」，助理直接基于当前展示的单词作答，无需用户重复说明。

---

### 3.4 System Prompt 动态构成

每次对话请求的 System Prompt 由以下四层拼接而成：

| 层级 | 内容 |
|------|------|
| 第一层：全局人设 | 助理名字、用户自定义的背景提示词（如「严格督促我」） |
| 第二层：全局记忆 | 从记忆系统中检索相关记忆条目，注入为「你已知道的关于用户的信息」 |
| 第三层：当前技能 Prompt | 根据当前页面所属技能，注入对应技能的专属 System Prompt |
| 第四层：页面上下文 | 当前页面的 JSON 上下文，帮助助理理解用户正在看什么 |

> **注意**：记忆注入采用检索而非全量注入，避免 Token 超限。检索策略为语义相似度 + 最近时间加权。

---

### 3.5 对话结束后的记忆提取

每次对话结束（用户关闭对话框或超过 5 分钟无新消息）后，系统异步执行记忆提取任务：

1. 将本次对话内容发送给 LLM，使用专用的记忆提取 Prompt
2. LLM 返回结构化记忆条目数组（JSON 格式）
3. 对新条目与已有记忆进行去重和合并判断
4. 写入记忆数据库，标记来源技能和时间

**记忆提取 Prompt 核心要求：**

- 只提取有长期价值的偏好、习惯、事实，不记录流水账
- 每条记忆用一句自然语言描述，不超过 30 字
- 标注置信度（0.0~1.0），低于 0.6 的不写入
- 返回格式：`[{ type, content, confidence, skill_source }]`

## 模块四：记忆系统

### 4.1 记忆数据结构

| 字段 | 说明 |
|------|------|
| `id` | 唯一标识符 |
| `content` | 自然语言描述，例如「用户更喜欢通过例句学单词」 |
| `type` | 记忆类型（见 4.2） |
| `skill_source` | 来源技能，例如 `english_learning` / `cooking` / `chat` |
| `confidence` | 置信度 0.0~1.0，低于 0.6 不展示给用户 |
| `created_at` | 首次记录时间 |
| `updated_at` | 最近更新时间 |
| `is_user_edited` | 用户是否手动修改过此条记忆 |

---

### 4.2 记忆类型分类

MVP 阶段定义六种记忆类型：

| 类型 | 示例 |
|------|------|
| `preference` | 「喜欢例句而不是直接背释义」 |
| `habit` | 「通常在晚上 10 点学习」 |
| `fact` | 「职业是软件工程师」 |
| `weakness` | 「对让步连词掌握不好」 |
| `progress` | 「已掌握 300 个四级词汇」 |
| `personality` | 「倾向于快节奏，不喜欢太多解释」 |

---

### 4.3 记忆页面设计

用户通过首页右上角菜单「记忆」入口进入，页面包含：

- **顶部标签栏**：「全部」+ 按日期过滤（显示最近 7 天有记忆写入的日期）
- **记忆卡片列表**：每张卡片显示记忆内容 + 来源标签（如「来自：背单词」）
- **编辑与删除**：点击卡片进入编辑页，支持修改内容或删除
- **手动添加**：卡片列表顶部的「+ 添加记忆」按钮，用户可主动补充助理不知道的信息

记忆页的设计原则是**「透明可控」**——用户能清楚看到助理「知道了什么」，并且有能力纠正错误记忆，建立对助理的信任。

---

### 4.4 记忆与对话的联动

当助理在对话中基于记忆作出判断时，轻量提示用户：

> 助理说：「根据你之前的习惯，我给你推荐了晚间复习计划。」
>
> 消息旁显示小标签「基于记忆」，点击可跳转到对应记忆条目。
>
> 用户如果觉得助理判断有误，可在对话中说「这个记忆不准确」，助理引导用户去修改。

## 模块五：通知与主动触达

### 5.1 主动触达机制

| 字段 | 说明 |
|------|------|
| 触达形式 | Push 通知（Firebase Cloud Messaging） |
| 触达时机 | 用户设定的推送时间段内，由后台任务决定是否发送 |
| 触达内容 | 基于用户当日状态生成，不是固定模板 |
| 默认频率 | 每日最多 1 次主动推送，避免打扰 |
| 智能抑制 | 当日已有对话的情况下不再主动推送 |

---

### 5.2 推送内容生成逻辑

后台任务每天在用户设定的推送时间窗口前 30 分钟执行：

1. 读取用户今日各技能使用情况（是否已完成每日目标）
2. 读取相关记忆，了解用户历史偏好
3. 调用 LLM 生成个性化推送文案（不超过 30 字）
4. 发送 Push 通知，用户点击后直接打开对应技能页 + 唤起对话框

**推送文案示例：**

> 「今天还差 5 个单词就完成目标了，要不要现在搞定？」
>
> 「你上次说想学做红烧肉，我帮你找了个简单版食谱。」
>
> 「有个你可能不知道的用法，关于上次学的 albeit——来看看？」

---

### 5.3 通知权限策略

- 首次安装**不请求**通知权限
- 用户完成第一次完整对话后，助理在对话中自然地询问：「我可以在你需要时主动提醒你吗？」
- 用户同意后才调起系统通知权限弹窗
- 用户拒绝则降级为应用内 Badge 提醒，不影响核心体验

## 附录：MVP 验证指标

### 核心留存指标

| 指标 | 目标 |
|------|------|
| Day 1 留存 | ≥ 60%（基准：普通工具类 App 约 40%） |
| Day 7 留存 | ≥ 40%（**核心验证指标**） |
| Day 30 留存 | ≥ 20% |

### 记忆系统先行指标

| 指标 | 目标 |
|------|------|
| 人均记忆条目（Day 7） | ≥ 15 条，说明用户在持续使用 |
| 记忆编辑率 | 5%~15%，用户在修改记忆说明他们关注并信任这个功能 |
| 记忆页访问频次 | 每周人均 ≥ 2 次 |

### 对话质量指标

| 指标 | 目标 |
|------|------|
| Push 回复率 | ≥ 30%，低于此说明推送内容或时机有问题 |
| 每日对话轮次 | 人均 ≥ 5 轮 |
| 技能覆盖率 | Day 7 用户平均使用 ≥ 2 个技能 |

### 种子用户计划

上线后第一批找 20~30 个种子用户，使用 Expo Go 分发测试包，跳过上架流程。观察两周数据后决定是否正式发布。重点访谈 Day 7 仍在使用的用户，了解他们感知到的核心价值。
